## Approach Summary (L2 Story)

`HybridRetrieverRouter`가 fusion → rerank 후 후보 품질을 자체 판단할 수 없는 구조에서, 검색 결과 품질을 평가하고 후속 동작(재질의 / fallback / abstain)을 분기시키는 `RetrievalEvaluator` SPI를 도입한다. SPI는 `core` 모듈에 위치하고, 기본 구현(score-threshold 기반 NoopOrPassThrough)을 제공한다. `HybridRetrieverRouter`에 평가 단계를 추가하여 평가 결과(`Decision`)를 반환·이벤트로 노출하고, autoconfigure에서 `@ConditionalOnMissingBean`으로 SPI override를 허용한다. CRAG/Adaptive RAG의 토대를 마련하되 본 작업에서는 분기 hook 표면만 노출한다 — 실제 corrective loop 호출은 후속 Story.

## Architecture

```
core/spi/
  RetrievalEvaluator.java        (신규 SPI — input: query+candidates, output: Decision)
  RetrievalEvaluator.Decision    (record: action, score, reason)
core/event/
  RetrievalEvent.EvaluationCompleted   (신규 record)

retrieval/
  HybridRetrieverRouter          (수정 — rerank 후 evaluator.evaluate 호출, decision 이벤트 발행)
  PassThroughRetrievalEvaluator  (신규 기본 구현 — 항상 ACCEPT)

autoconfigure/
  AgenticRagRetrievalAutoConfiguration  (수정 — RetrievalEvaluator @ConditionalOnMissingBean Bean,
                                          HybridRetrieverRouter 생성자에 주입)
  AgenticRagProperties                  (수정 — retrieval.evaluator.{enabled,min-score} 추가)
```

의존 방향 유지: `retrieval` → `core` only. autoconfigure만 조립. SPI는 결정만 반환하고 실제 분기 액션(재질의 등)은 호출자(현재 router, 향후 orchestrator)가 책임진다.

## Implementation Plan

| # | 파일 | 변경 유형 | 규모 | 요약 |
|---|------|---------|---|------|
| 1 | `agentic-rag-core/.../common/spi/RetrievalEvaluator.java` | 신규 | S | SPI 인터페이스 + `Decision` record (`action`, `score`, `reason`, `attributes`) + `Action` enum (`ACCEPT`, `RETRY`, `FALLBACK`, `ABSTAIN`) |
| 2 | `agentic-rag-core/.../common/event/RetrievalEvent.java` | 수정 | S | `EvaluationCompleted(action, score, evaluator, elapsedMillis, ...)` record 추가 |
| 3 | `agentic-rag-retrieval/.../evaluate/PassThroughRetrievalEvaluator.java` | 신규 | S | 항상 ACCEPT 반환하는 기본 구현 |
| 4 | `agentic-rag-retrieval/.../evaluate/ScoreThresholdEvaluator.java` | 신규 | S | top1 fused score < threshold → RETRY 반환 (선택 구현, 기본 비활성) |
| 5 | `agentic-rag-retrieval/.../HybridRetrieverRouter.java` | 수정 | M | 생성자에 `RetrievalEvaluator` 추가, rerank 후 `evaluate` 호출 → `Decision`을 결과 wrapper(`RetrievalOutcome`)로 반환하거나 메타데이터에 부착 + `EvaluationCompleted` 이벤트 발행 |
| 6 | `agentic-rag-retrieval/.../RetrievalOutcome.java` | 신규 | S | `record RetrievalOutcome(List<Document> documents, Decision decision)` — router 호출자가 분기 hook으로 사용 |
| 7 | `agentic-rag-autoconfigure/.../AgenticRagRetrievalAutoConfiguration.java` | 수정 | S | `RetrievalEvaluator` Bean (`@ConditionalOnMissingBean`, properties로 PassThrough/ScoreThreshold 선택), `HybridRetrieverRouter` 생성자에 주입 |
| 8 | `agentic-rag-autoconfigure/.../AgenticRagProperties.java` | 수정 | S | `Retrieval.Evaluator { enabled, strategy, minScore }` 중첩 properties |

**RetrieverRouter 인터페이스 변경 정책**: 기존 `retrieve(Query) : List<Document>` 시그니처는 **유지**. 신규 default 메서드 `retrieveWithDecision(Query) : RetrievalOutcome`을 추가하여 SPI 사용자 hook을 제공한다 — 기존 사용자 코드 비파괴.

## Key Decisions

| # | 결정 | 대안 | 선택 이유 |
|---|---|---|---|
| 1 | `RetrievalEvaluator`를 `core/spi`에 둔다 | `retrieval` 모듈에 둔다 | autoconfigure가 모듈 경계를 넘어 조립하려면 SPI는 `core`에 있어야 한다 (`module-plan-and-design.md` §4-1) |
| 2 | `Decision`은 결정만 반환, 분기 액션은 호출자 책임 | evaluator가 직접 재질의 호출 | SRP — evaluator는 판단만, 재시도/fallback은 router 또는 orchestrator의 정책 영역 |
| 3 | `RetrieverRouter` 인터페이스에 `retrieveWithDecision` default 추가 | 기존 `retrieve` 시그니처 변경 | 비파괴 변경 — Phase 1 코드 호환 유지 |
| 4 | 기본 구현은 `PassThroughRetrievalEvaluator` (항상 ACCEPT) | `ScoreThresholdEvaluator`를 기본 | Phase 2에서 corrective loop 도입 시 켤 수 있게 hook만 노출, 기본은 부작용 0 |
| 5 | `Action` enum 4종 (ACCEPT/RETRY/FALLBACK/ABSTAIN) | 단일 boolean grounded | CRAG·Adaptive RAG가 요구하는 분기 다양성을 미리 확보 — Phase 2 상위 작업이 enum 값을 그대로 소비 |

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| `RetrieverRouter` 인터페이스 확장이 외부 사용자 구현체를 깨뜨림 | M | default 메서드로만 추가, 기존 `retrieve` 시그니처 불변 |
| `Decision.score` 의미 모호 (fused score vs rerank score vs evaluator 자체 score) | L | Javadoc에 "evaluator 자체 0..1 confidence"로 못박고, 후보 score는 metadata로 별도 노출 |
| Phase 2 항목(CRAG)을 끌어들여 over-engineering | L | RETRY/FALLBACK 액션은 enum 정의만 — 실제 router 분기 로직은 ACCEPT/ABSTAIN만 1차로 처리, 나머지는 호출자 위임 |

## Open Items

- ScoreThresholdEvaluator 기본 활성화 여부 — 본 작업에서는 `enabled=false` 기본. Phase 2 corrective Story에서 결정.
- `RetrievalOutcome` vs `Decision`을 `RetrievalEvent.EvaluationCompleted`로만 노출하고 router 반환 타입은 그대로 둘지 — 현재 결정: outcome wrapper 추가. 사용자가 hook을 코드 경로에서 받지 않고 이벤트만으로 분기하기 어렵기 때문.
