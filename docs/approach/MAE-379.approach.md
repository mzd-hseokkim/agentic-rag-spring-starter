## L1 Single (5줄)

- **변경 영역**: `agentic-rag-autoconfigure/src/main/java/kr/co/mz/agenticai/core/autoconfigure/tools/fs/` 하위에 grep 전용 신규 파일 4개 — `GrepTool.java`(단일 `@Tool grep(...)` 메서드만 보유; sibling `FileSystemTools`와 분리하여 본 태스크의 surgical 범위 유지), `dto/GrepResult.java`(`outputMode`별 union — `List<String> files` / `List<MatchLine> contentLines` / `Map<String,Long> counts` + `truncated` flag), `dto/MatchLine.java`(path, lineNumber, line, kind: MATCH/BEFORE/AFTER), `enum/GrepOutputMode.java`(`FILES_WITH_MATCHES` / `CONTENT` / `COUNT`). 테스트는 동일 모듈 `src/test/.../tools/fs/` 하위 `GrepToolTest.java` 1파일에 outputMode별/contextLines/multiline/PatternSyntaxException/NFC-NFD 케이스 묶음. MAE-378이 만든 `FileSystemToolCallbackProvider`의 `toolObjects(...)` 인자에 `GrepTool` 인스턴스 1줄 추가는 MAE-380(autoconfigure) 책임이므로 본 태스크에서는 건드리지 않는다 — provider 시그니처 변경 없음. 이벤트 발행은 MAE-382 범위 외.
- **핵심 결정**: (1) sibling MAE-378의 tool 이름 컨벤션을 따라 `fs_grep`으로 노출 — `@Tool(name="fs_grep", description=...)`. (2) 엔진은 `java.util.regex.Pattern` + `Files.walk(sandbox.resolve(path))` 단일 패스 — 라인 단위 `Files.lines(p, StandardCharsets.UTF_8)`. `multiline=true`이면 파일 전체를 `Files.readString` 후 `Pattern.compile(pattern, Pattern.DOTALL | Pattern.MULTILINE).matcher(text)` 으로 전환(라인 경계 무시 매칭). `caseInsensitive=true`이면 `Pattern.CASE_INSENSITIVE | Pattern.UNICODE_CASE` 조합. (3) `glob`/`type` 필터는 진입 시 `PathMatcher`로 1차 walker 컷, 미지정 시 모든 regular file. (4) `outputMode`별 처리: `FILES_WITH_MATCHES`는 첫 매치 즉시 break하여 파일 경로만 수집 — `headLimit` 적용; `CONTENT`는 매치 라인 + `contextLines`(앞/뒤 동일 N라인) 슬라이딩 윈도우, 중복 라인은 lineNumber 기준 dedup, kind 표기로 BEFORE/MATCH/AFTER 구분; `COUNT`는 파일별 매치 개수 누적 후 0 카운트 제거. (5) regex compile 에러는 `PatternSyntaxException`을 `IllegalArgumentException("invalid regex: " + e.getDescription() + " at index " + e.getIndex())`로 wrap — LLM에게 fix 가능한 메시지 제공. (6) 한글 파일명은 sandbox 진입 직후 `Normalizer.normalize(path, Form.NFC)`로 정규화하여 macOS NFD 경로/Windows NFC 경로 모두 동일 키로 비교. (7) 모든 path 인자는 `sandbox.resolve(rawPath)` 통과 후만 사용 — 직접 `Paths.get` 금지. (8) 출력 상한은 MAE-377의 `OutputLimits.maxResults`(50)/`maxBytes`(4KB)/`maxChars`(500)를 `applyList`/`applyContent` 헬퍼로 적용, 초과 시 결과 객체의 `truncated=true` + 마지막 항목/라인에 `(truncated, N more)` append.
- **검증**: `./gradlew :agentic-rag-autoconfigure:test --tests "*GrepToolTest*"`. 테스트 케이스 — (a) `outputMode=FILES_WITH_MATCHES` 매치 파일 N건 반환 + 0매치 빈 List, (b) `outputMode=CONTENT` 기본/`contextLines=2`(`-A -B -C` 등가 — 앞뒤 2라인 각각 BEFORE/AFTER kind로 부착), (c) `outputMode=COUNT` 파일별 카운트 + 0건 파일 미포함, (d) `multiline=true` 줄바꿈 가로지르는 패턴(`foo\nbar` 등) 매치, (e) `caseInsensitive=true` 대소문자 무시 매치, (f) `[`처럼 깨진 regex → `IllegalArgumentException` + 메시지에 위치/원인 포함, (g) 한글 파일명 NFC/NFD 양쪽 표기를 같은 결과로 노출, (h) `headLimit` 초과 입력의 `truncated=true` + `(truncated, N more)` 표기 검증, (i) sandbox 거부(`../etc/passwd`) → `SecurityException` 또는 sandbox 정의된 예외 그대로 propagate.
- **리스크**: (1) MAE-377의 `OutputLimits` / `WorkspaceSandbox` API 시그니처 미확정 — sibling 머지 후 import 정합성 1회 점검 필요. 우회: 둘 다 미머지 상태면 본 태스크는 자체 `GrepLimits`/`SandboxAdapter` 임시 인터페이스로 컴파일 유지 + MAE-377 머지 시 1회 rename. (2) `multiline=true` + 큰 파일은 `Files.readString` 전체 적재 — `maxBytes` 상한을 파일 크기에 선검사 후 초과 시 라인 모드로 fallback + warning. (3) `headLimit` 기본값을 `OutputLimits.maxResults`에 맞추면 `CONTENT` 모드에서 contextLines 포함 후 50 라인을 초과할 수 있음 — 컨텍스트 포함 후 절단 적용 일관성 유지. (4) `Files.walk` 심볼릭 링크는 sandbox가 차단(`LinkOption.NOFOLLOW_LINKS`) — 본 태스크 추가 방어 불필요(MAE-377 책임).
- **롤백**: 본 태스크는 grep 전용 신규 파일만 추가 — `GrepTool.java` + `dto/GrepResult.java` + `dto/MatchLine.java` + `enum/GrepOutputMode.java` + 테스트 1파일 삭제로 원복. 기존 파일 수정 0건이므로 revert는 PR 단위 단순 revert.
