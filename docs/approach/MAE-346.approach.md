# Approach: MAE-346 - 메모리 정책 SPI — 토큰 윈도우 / Rolling Summary

- **Level**: L2 (Story)
- **Source**: N/A — discover 생략 (Jira description 직접 사용)

## Approach Summary (L2 Story)

`MemoryStore`(raw 저장)와 메시지 trimming 정책을 분리하기 위해 `MemoryPolicy` SPI를 도입한다. `MemoryPolicy.apply(history, query, systemPrompt)`가 LLM에 전달할 `List<Message>`를 반환하며, 기본 구현 3종(`RecentMessagesPolicy` / `TokenWindowPolicy` / `RollingSummaryPolicy`)을 `@ConditionalOnMissingBean`으로 제공한다. `SummaryAgent`는 `historyLimit` 직접 호출을 제거하고 정책을 통해 메시지를 받는다. Spring AI `ChatMemory`는 **어댑터 방식**으로 상호 운용 — 기본은 자체 SPI이며 사용자가 ChatMemory bean을 노출하면 Spring AI 생태계와 결합 가능.

## Architecture

```
agentic-rag-core/common/spi/
  MemoryStore.java                  (기존, 변경 없음)
  MemoryPolicy.java                 (신규 SPI)
  TokenCounter.java                 (신규 SPI — TokenWindowPolicy 의존)

agentic-rag-core/common/memory/
  MemoryRecord.java                 (기존)
  policy/
    RecentMessagesPolicy.java       (신규, 기본)
    TokenWindowPolicy.java          (신규)
    RollingSummaryPolicy.java       (신규, ChatModel 주입)
    HeuristicTokenCounter.java      (신규, char/4 추정 기본 구현)
    SpringAiChatMemoryAdapter.java  (신규 — Spring AI ChatMemory ↔ MemoryStore 브릿지)

agentic-rag-agents/
  SummaryAgent.java                 (수정 — MemoryPolicy 사용)

agentic-rag-autoconfigure/
  MemoryAutoConfiguration.java      (수정 — Policy/TokenCounter 등록 + properties)
```

의존 방향: `agents` → `core(spi+memory.policy)`. `RollingSummaryPolicy`는 `ChatModel`을 생성자 주입 (요약용 LLM은 사용자 application의 기본 ChatModel 재사용).

## Key Decisions

| # | 결정 | 대안 | 선택 이유 |
|---|---|---|---|
| 1 | `MemoryPolicy.apply(List<MemoryRecord> history, String query, String systemPrompt) → List<Message>` | `List<MemoryRecord>` 반환 / Builder DTO | LLM 직전 단계이므로 Spring AI `Message`가 자연스러움. systemPrompt 인자는 RollingSummary가 SYSTEM 메시지에 요약을 병합하기 위해 필요 |
| 2 | Spring AI `ChatMemory`와는 **어댑터 공존** (자체 SPI 유지) | ChatMemory로 대체 | 기존 `MemoryStore` SPI/구현체(InMemory/Redis)와 호환 유지. ChatMemory는 사용자 옵션으로 `SpringAiChatMemoryAdapter` 통해 연결 |
| 3 | TokenCounter는 별도 SPI + Heuristic 기본 (char/4) | jtokkit/Spring AI tokenizer 의존 | starter는 모델 비종속. 정확도 필요 시 사용자가 jtokkit 기반 빈 등록 |
| 4 | RollingSummary 트리거 = `tokenBudget` 초과 시 oldest 절반(또는 `summarizeFraction`)을 한 번에 요약 | 매 N턴 주기적 요약 / 메시지 수 기준 | 토큰 폭발 방지가 본질 목표 — budget이 직접 지표. 절반 단위로 압축해 빈번한 LLM 호출 회피 |
| 5 | RollingSummary의 SYSTEM 병합 = 기존 systemPrompt 뒤에 `\n\n[Conversation summary so far]\n<요약>` append, 별도 SystemMessage 추가하지 않음 | 별도 SystemMessage append | Spring AI 다수 모델이 단일 SYSTEM만 신뢰. 요약은 누적되지 않고 최신본만 유지(이전 요약+신규 청크를 다시 합쳐 재요약) |
| 6 | `historyLimit` 호환: `RecentMessagesPolicy`가 기본값으로 등록되며 동일 동작 | 즉시 deprecate | 사용자 설정 무중단 마이그레이션 |

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| 토큰 카운트 부정확 (Heuristic char/4) → budget 위반 | 중 | budget에 안전 마진(예: 0.85) 적용 + 정확 카운터 SPI 교체 가능 문서화 |
| RollingSummary 요약 LLM 호출이 응답 latency에 추가됨 | 중 | trigger를 budget 초과 시점으로 한정 + 비동기 사전 요약은 Phase 2 |
| Spring AI `ChatMemory` API 변동 | 저 | 어댑터 단일 클래스에 격리 — 변경 시 한 파일만 수정 |
| `MemoryPolicy.apply` 시그니처가 systemPrompt를 받는 것은 추상화 누수 | 저 | RollingSummary 요구사항이 명확 — 대안(Policy가 SystemMessage를 내장)은 더 큰 누수. trade-off 수용 |
