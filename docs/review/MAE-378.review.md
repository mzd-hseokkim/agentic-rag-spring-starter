### Info

1. **FileSystemTools.java L82 — glob truncation 미`OutputLimits.apply` 우회**
   - glob은 inline loop로 sentinel을 `String`으로 append. listDir은 `limits.apply(entries, sentinel)` 호출. 두 경로의 truncation 처리 방식이 다름.
   - 영향: 기능적으로 동일하나 일관성 저하. `OutputLimits` API가 MAE-377에서 정식화될 때 통합 검토.

2. **OutputLimits.java L56-68 — `applyString` 미사용**
   - 본 task 내 호출처 없음. `readFile`은 stream 기반이라 `applyString(content, totalLines)`의 `totalLines` 사전계산이 어색 → 자체 truncation 로직 채택.
   - 권장: MAE-377 정식화 시 stream-friendly 시그니처로 재설계하거나 dead method 제거.

3. **FileSystemTools.java L130-133 — listDir sentinel 객체 사전 생성**
   - `entries.size() - limits.maxResults()`가 음수일 수 있는데 (truncation 미발생 시) sentinel `DirEntry`가 무조건 생성됨. `apply` 내부에서 size 비교 후 폐기.
   - 영향: 미미. `apply` 시그니처를 `Function<Integer, T> sentinelFactory`로 받게 향후 변경 권장.

4. **FileSystemTools.java L62 — `Files.walk` 심볼릭 링크 정책**
   - 기본값 = symlink 미추적(loop 위험 낮음) + sandbox가 `../` 거부 → 외부 escape 차단. OK. (sandbox.resolve 자체가 `normalize().startsWith(root)` 검증을 placeholder TestWorkspaceSandbox에서 수행, MAE-377 정식 구현에서도 동일 보장 필요.)

5. **FileSystemTools.java L70-76 — mtime 정렬 IO 비용**
   - approach 리스크에 명시 (headLimit=50 자연 제한, Phase 2 캐싱은 금지). 현행 acceptable.

6. **변경사항 미커밋 상태**

### Security

- 모든 외부 입력 경로는 진입 직후 `sandbox.resolve()` 통과. test로 `../` 거부 양방향 검증 (`FileSystemToolsGlobTest#sandboxRejectsPathTraversal`, `FileSystemToolsReadFileTest#sandboxRejectsPathTraversal`).
- credential/secret 하드코딩 없음.
- LLM 입력 → `PathMatcher("glob:" + pattern)` injection 위험: `PathMatcher`는 glob DSL만 해석, OS command 실행 없음. 안전.
- `Files.lines` 기본 UTF-8 — 비UTF-8 파일은 `MalformedInputException` 발생 → `UncheckedIOException`으로 래핑되어 LLM에 노출. 운영에서 binary 파일 호출 시 noisy할 수 있으나 Phase 1 범위 외.
