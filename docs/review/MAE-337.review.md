### Info

- **I1** `HybridRetrieverRouter.java:78` `retrieveWithDecision`이 `RetrieverRouter` SPI 인터페이스가 아닌 구현 클래스에만 존재. Approach Key Decision #3은 "RetrieverRouter 인터페이스에 default 메서드 추가"라고 명시했으나 실제로는 구현 클래스 메서드로만 노출됨. 호출자가 `RetrieverRouter` 타입 변수로 받으면 hook 사용 불가. 다만 비파괴 변경 목표는 달성되었고, 인터페이스에 추가하면 외부 구현체(없는 것으로 보이지만)를 깰 가능성이 있어 보수적 선택으로 해석 가능. **후속 Story에서 SPI 표면 결정 필요**. (file: `agentic-rag-core/.../common/spi/RetrieverRouter.java`)
- **I2** `ScoreThresholdEvaluator.java:53` `extractFusedScore`는 `FUSED_SCORE` 키 부재 시 0.0으로 처리하여 RETRY를 유도. 의도적 설계지만 metadata 키 자체가 fusion 단계에서 항상 채워지는지 호출 컨텍스트 확인 필요 (현재 NoopReranker / RRF 경로에서는 채워짐).
- **I3** `Decision.java` `score` 필드 의미가 evaluator-confidence vs document-score로 혼동될 수 있음. Javadoc은 "evaluator's own confidence"로 명시하지만 `ScoreThresholdEvaluator`는 top1 fused score를 그대로 `score`로 반환하여 약속을 살짝 어긴다. 후속 corrective Story에서 개별 evaluator마다 의미 일관성 점검 권장.
- **I4** `EvaluationCompleted.evaluator` 필드는 `getClass().getSimpleName()` 문자열. 외부 사용자가 instance type-id로 사용하기 어렵다 — 하지만 현 단계는 관찰성 목적이므로 충분.

### Positive Notes

- SPI 위치(`core`) / 기본 구현 위치(`retrieval`) / 조립(`autoconfigure`) 분리가 모듈 경계 정책을 정확히 따른다.
- `Decision` record의 컴팩트 컨스트럭터에서 `score` 범위 검증과 `attributes` immutable copy를 동시에 수행 — 방어적 설계 합리적.
- T4 (Mockito spy로 `evaluate` 1회 호출 검증) + T6b (RRF 1/61 score < 0.5 RETRY 경로)가 router/evaluator 통합을 단단히 cover.

- Approach의 "Phase 2 over-engineering 회피" Risk를 잘 지킨다 — RETRY/FALLBACK/ABSTAIN action enum은 정의만, router는 ACCEPT 외에는 호출자에게 위임.
