### Info

1. **`rag.ts:39` — `Completed` payload 형태 가정**
   백엔드 `RagStreamEvent.Completed`는 record `Completed(RagResponse response)`로 직렬화되므로 `{ response: RagResponse }` 가정은 Jackson default record 직렬화 기준 맞음. 다만 백엔드 SSE writer가 record를 어떻게 unwrap하는지 첫 수동 테스트로 확인하면 안전.

2. **`rag.ts:44` — `Failed` 이벤트 payload 미사용**
   백엔드 `Failed(Throwable error)`의 SSE JSON 형태(throwable이 그대로 직렬화될 가능성)는 클라이언트에서 파싱되지 않음. 현재는 `es.close()`만 호출하므로 사이드이펙트 없음. 추후 에러 메시지 노출이 필요해지면 백엔드에서 `Failed`를 `{ message: string }` 형태로 매핑하는 작업이 함께 필요.

3. **`client.ts:10` — multipart 호출에서 Content-Type 미지정 처리**
   `fetchJson`은 항상 `application/json`을 강제하므로 multipart 경로는 별도 `fetchMultipart`로 분리 — 의도된 분리이며 주석으로 명시됨. OK.

4. **`actuator.ts:3-13` — Actuator 스키마 수기 정의**
   Spring Boot Actuator 표준 응답에 맞춰 직접 정의. 향후 Actuator 응답 형태가 바뀌어도 demo 한정 영향이라 수용 가능.
