### Info

1. **GrepTool.java L307 — `count()` 결과 키가 절대 경로 (`file.toString()`)**
   - `filesWithMatches`/`content`는 `relativize(base, file)` 사용. COUNT 모드만 절대 경로 키 사용 → LLM이 받는 결과의 path 포맷이 mode별로 다름.
   - 영향: 기능 정상이지만 UX 일관성 저하. 한 줄 변경(`file.toString()` → `relativize(base, file)`)으로 해소 가능. 추후 정리 권장.

2. **GrepResult.java L38 — `ofCounts(...)`는 `truncated` 항상 false**
   - COUNT 모드는 `headLimit`을 무시하고 전체 파일 카운트. approach "(8) 출력 상한 적용" 의도와는 일부 어긋남.
   - 영향: COUNT 결과가 매우 큰 워크스페이스에서 컨텍스트 과다할 수 있음. 다만 카운트는 라인이 아니라 파일 단위라 실제 영향 낮음. 후속(MAE-380 or 보안 테스트 MAE-381)에서 다룰지 결정.

3. **GrepTool.java L177 — FILES_WITH_MATCHES truncation sentinel 형식이 sibling과 다름**
   - sibling MAE-378 `glob`은 인라인 `String` append, `listDir`은 `OutputLimits.apply`. 본 태스크는 인라인 append. MAE-377 머지 후 통합 검토.

4. **GrepTool.java L212 — CONTENT multiline 분기 `text.split("\n", -1)`**
   - Windows CRLF 파일에서 `\r`가 라인 끝에 남음 (`MatchLine.line` 출력에 포함). 매칭/카운트는 영향 없음(regex가 `\r` 무시) — 출력 가독성만 미세 영향. 후속 정리 권장.

5. **OutputLimits.maxBytes / maxChars 미존재 — approach (8) 표기와 불일치**
   - approach에는 `OutputLimits.maxResults(50)/maxBytes(4KB)/maxChars(500)` 언급. 실제 placeholder는 `maxResults`, `maxLines`만. 본 구현은 `maxResults`만 사용하고 4MB 자체 가드(`readStringSafe`)로 대신함. MAE-377 정식 API 머지 시 정합성 점검 대상 — approach Open Item에 이미 기록됨.
